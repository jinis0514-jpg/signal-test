# 결제 안정성 점검 체크리스트

## 현재 상태 (코드 기준)

- `subscriptionService`는 mock 기반 플랜 반영 로직 포함
- `checkoutIntent`는 결제 의도 객체 생성 수준
- 실결제 웹훅/서버 확정 플로우는 미완성

## 출시 전 필수 아키텍처

1. 결제 생성: 서버에서 결제 세션 생성
2. 결제 확정: PG/Stripe 웹훅 수신
3. 상태 반영: 서버가 `subscriptions` 업데이트 (idempotent)
4. 프론트 동기화: 주기/이벤트로 최신 상태 반영

## 체크 항목

## 1) 성공/실패 처리

- [ ] 결제 성공 시 플랜 즉시 반영
- [ ] 결제 실패 시 기존 플랜 유지 + 사용자 안내
- [ ] 결제 대기/중단 상태 명확히 표시

## 2) 중복 결제 방지

- [ ] `idempotency_key` 사용
- [ ] 같은 `checkout_session_id` 중복 처리 차단
- [ ] 다중 탭 결제 시 race condition 방지

## 3) 웹훅 동기화

- [ ] 웹훅 서명 검증 필수
- [ ] 웹훅 재시도 시 중복 반영 방지
- [ ] 반영 실패 시 DLQ/재처리 큐 존재

## 4) 환불/취소/만료

- [ ] 환불 시 즉시 또는 정책 시점에 플랜 하향
- [ ] 사용자가 취소한 경우 `status=canceled` 반영
- [ ] 만료 크론/스케줄러로 `expired` 전환

## 5) 권한 안전성

- [ ] 사용자가 직접 `subscriptions.plan`을 올릴 수 없도록 서버 단일 경로화
- [ ] RLS와 별개로 서버에서 결제 상태 검증 후 업데이트

## 결제 장애 대응 체크

- 결제 provider 상태 페이지 확인
- 웹훅 수신 로그 확인
- 최근 15분 결제 성공/실패 비율 확인
- `subscriptions` 반영 지연 건수 확인
